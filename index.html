This is chess's superpower: Highlights don't need manual tagging. A Stockfish/Leela eval pass on the PGN produces a fully annotated highlight reel automatically. This is more deterministic than any other sport on the WSC platform.

♜ Sport Fundamentals

Chess match structure, time controls, scoring, and key rule variants

Match Format

  • 1v1 (one player per side)
  • Pieces: King, Queen, Rooks, Bishops, Knights, Pawns
  • White moves first, players alternate
  • Win: Checkmate opponent's King
  • Draw: Stalemate, repetition (3x), 50-move rule, agreement, insufficient material
  • Format: single game, match series (best of N), or round-robin tournament

Time Controls

  • Classical: 90 min + 30 sec/move (FIDE standard)
  • Rapid: 15–25 min + 10 sec/move
  • Blitz: 5 min + 3 sec/move
  • Bullet: 1–2 min per player
  • Armageddon: White 5 min, Black 4 min — draw = Black wins. Decisive by rule.

Scoring note: Chess uses points, not goals. A win = 1 pt, draw = 0.5 pt each. In tournaments, cumulative points determine standings. The WSC score model needs to support decimal/half-point scoring.

Unlike Padel (which shares Tennis scoring), Chess has no scoring, no physical actions, and no video event model in common with any WSC sport.

Bottom line: Chess is not "racquet sport without a ball." It's a fundamentally different content type. A sub-sport approach would require overriding 100% of the parent logic — defeating the purpose entirely. If Chess is onboarded, it must be a new sport.

For Chess, WSC's traditional content stack (Clipro → video clip → publish) would need a parallel output type:

  • Board animation renderer: FEN → animated SVG/canvas → MP4

Why start here: Board animation requires zero live video infrastructure. Can be built as a standalone microservice. Proves the content value to a client without the full WSC sport onboarding overhead.

  • New SystemTypeId for Chess
  • Resolver: ChessResolver with move-based game state (FEN, eval, clock)
  • Actions: Brilliant, Blunder, Checkmate, Sacrifice, Promotion, TimePressure, DecisiveSwing
  • Game progress: Move number, eval score, clock, material count, phase (Opening/Middlegame/Endgame)
  • Ingestion adapter: Chess.com WebSocket or DGT relay
  • Integration with eval engine (Stockfish on WSC infra or cloud API)

Dev estimate (internal reference, Shahar Cohen): ~6 weeks neto + 2 weeks for new socket = ~8 weeks neto / ~3–4 months calendar

Phase 3 — Face-Cam Video Integration (2–3 sprints)

  • Negotiate video stream + timestamp sync access with Chess.com or FIDE broadcast team
  • Build video sync layer: move timestamp → video timecode → face-cam cut
  • Compose: board animation + face-cam overlay = full broadcast-quality highlight
  • This is the premium tier — high visual impact, most similar to WSC's traditional output
TL;DR — Full standalone Chess sport type. ~8 weeks neto / ~4 months calendar. New content paradigm (board animation). High but justified investment if Chess.com or FIDE signs.

Solution A — Chess as a New Sport

New SystemTypeId Full Sport Stack Eval-Driven Highlights

Solution A — Pros & Cons

Trade-off analysis for building Chess as a standalone sport

✅ Pros

  • Automation advantage — Eval-based highlight detection is more deterministic than most sports. No manual tagging needed.
  • PGN universality — One parser serves all chess providers (Chess.com, Lichess, DGT, FIDE)
  • No video dependency for MVP — Board animation works without live video infrastructure
  • Clean architecture — Chess is genuinely unique; forcing it under another sport would be architecturally wrong
  • Future platform — Opens "mind sports" category (Go, Checkers, Bridge, Poker)
  • Chess audience is digital-native — Social clip appetite is huge (Chess.com has 1B+ views/month)
  • Reasonable effort — 8 weeks neto is comparable to a Tennis sub-sport

❌ Cons

  • New content paradigm — Board animation renderer doesn't exist in WSC stack. New component to build and maintain.
  • Eval engine dependency — Stockfish integration adds infra complexity. Needs compute resources.
  • No signed clients — Full build without confirmed revenue is risky
  • Chess.com self-sufficiency — Biggest potential client already has in-house content team. WSC value-add must be automation at scale.
  • Face-cam requires negotiation — Video sync access from Chess.com/FIDE broadcast is not guaranteed
  • Niche market — Chess client base is likely smaller than Soccer or Basketball
TL;DR — Manual board animation MVP: build a lightweight PGN → animation tool, hand it to an analyst. Zero sport infrastructure. Validates demand and delivers content before any platform investment.

Solution B — Manual / Board Animation MVP

No Sport Type PGN-only Analyst-operated

Approach Overview

Skip the WSC platform entirely for now. Build a standalone lightweight tool: feed it a PGN file, it outputs annotated board animation clips. An analyst operates it to produce content. No Resolver, no SystemTypeId, no MovieCreator integration.

TL;DR — Fastest path to content with lowest risk. Doesn't scale to real-time or multi-sport, and can't leverage WSC's distribution network without platform integration.

Solution B — Pros & Cons

Trade-off analysis for the manual board animation MVP

✅ Pros

  • 2–4 sprints vs 6–8 — Half the investment of a full sport type
  • Validates demand — Can show a client real content before building the platform
  • Low operational cost — PGN processing is near-instant. Analyst time = minutes/game.
  • Uses open-source chess libraries — chess.js, Stockfish WASM — no reinvention
  • Zero risk if market doesn't materialize — Standalone tool, not baked into core platform
  • Reusable renderer — Board animation engine built here feeds Phase 2 (Solution A)

❌ Cons

  • No face-cam video — Board animation only. Premium broadcast-quality content needs Phase 2.
  • Tech debt — Standalone tool will need rework when integrating with the platform
TL;DR — Chess was estimated internally at 8 weeks neto. AFL (new sport) took ~3 months. Athletics (multi-sub-sport) took 10-12 months. Chess is closer to AFL in complexity but with a simpler event model.

🎯 Recommendation

Pragmatic path forward with graduation criteria

Phased Approach: Board Animation MVP → Full Chess Sport

Chess is genuinely promising — strong digital audience, excellent data richness, and algorithmic highlight detection that makes automation easier than most sports. But with no signed clients and a new content paradigm (board animation), the right move is to validate before building the full platform.

  1. Phase 1 — Now: Board Animation MVP (B) — 2–4 sprints. PGN parser + key moment detection + standalone board renderer. No platform integration. Can demo real chess highlights to Chess.com or FIDE in 6–8 weeks.
  2. Phase 2 — On deal signature: Full New Sport (A) — 6–8 additional sprints. Add ChessSystemTypeId, resolver, WSC ingestion, real-time pipeline, and platform distribution. Board renderer from Phase 1 is reused — not thrown away.
  3. Phase 3 — With broadcast partner: Face-Cam Video — 2–3 sprints. Negotiate timestamp sync with Chess.com/FIDE. Add face-cam overlay. Premium output tier.
